The shape of a comment

Post #2 — by the BAM team

A bam-comments message is one ERC-8180 message. Its contents field begins with the 32-byte ERC-8179 contentTag (keccak256(utf8("bam-comments.v1"))) so the on-chain index keeps every comments app's stream isolated from every other app sharing the same Poster.

After the tag prefix, the bytes are app-opaque. The widget packs them as: a one-byte version, a one-byte kind (comment or reply), the 32-byte postIdHash derived from (siteId, postId), a uint64 timestamp, (optionally) the parent message hash for replies, then a length-prefixed UTF-8 body. Length-prefixing the siteId/postId bytes when deriving postIdHash keeps ("ab","cd") from colliding with ("abc","d").

Threading on the read side is a flat-list-to-tree pass with orphan and cycle filtering, then display depth clamped to two so the UI stays readable. The wire-level depth is preserved — clients with different opinions about display can still walk the full chain.
